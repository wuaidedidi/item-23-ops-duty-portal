# 全量代码与业务逻辑双重审查循环

本工作流执行 **审查 → 逻辑推演 → 修复 → 对抗验证** 的全自动闭环循环。
**核心原则：全程自动执行，不停下来询问用户确认。不仅保证代码不报错，更要保证业务逻辑符合常识（如：数量不可为负、非管理员不可见后台）。发现问题后直接按通用最佳实践修复，修复完直接验证，验证不过直接继续修。**

---

## 前置准备

// turbo
1. 先阅读当前会话中的要求，提取 **完整功能清单**，列出每一条需要实现的功能点
// turbo
2. 用 `todo_list` 工具创建审查清单，每个功能点一条，初始状态全部为 `pending`
// turbo
3. **业务逻辑推理**（根据项目类型自动推导）：
   - 根据功能名推导隐含业务规则（例如："购物车" → 隐含 "库存不足不可加购"、"数量必须>0"；"对战" → 隐含 "不能和自己对战"、"掉线判负"）
   - 识别系统中的 **角色与权限**（Admin vs User、已登录 vs 未登录）
   - 识别关键 **状态机**（如：房间状态 waiting→playing→finished 的流转顺序）
   - 将推导出的隐含规则也加入审查清单
// turbo
4. 用 `find_by_name` 和 `code_search` 扫描项目结构，建立 **文件→功能** 的映射关系

---

## 第一轮：全量代码与业务逻辑审查

**原则：不允许偷懒，完全无遗漏地查看每一个源码文件。像测试工程师一样思考——不仅看代码是否写了，更要看逻辑是否闭环**

对项目中的 **每一个** JS/TS/源码文件，按以下步骤逐文件审查：

// turbo
4. 使用 `read_file` 工具 **完整阅读** 当前文件的全部代码（如果文件超过1000行，分段读取直到读完）
// turbo
5. 对照功能清单，进行 **双重检查**：

   **A. 代码级检查（基础）**
   - 该文件声明的功能是否 **真正实现**（不是空函数、不是TODO占位）
   - 函数调用的参数、ID、事件名是否与其他模块 **一致匹配**
   - 数据流是否完整：输入 → 处理 → 输出 → UI更新 → 存档，每一环是否都接通
   - 事件发射(emit)与监听(on)的事件名是否完全匹配
   - 跨模块引用的函数/方法是否真实存在于目标模块的公开接口中
   - UI视图是否有入口可以访问（导航栏/按钮是否存在）

   **B. 业务逻辑检查（进阶）**
   - **边界值防护**：输入是否有 min/max 限制？后端是否校验负数/零/超大值？
   - **权限与可见性**：敏感操作（删除、配置、管理）是否有权限守卫？未登录用户能否绕过路由访问受保护页面？
   - **空状态处理**：列表为空时是否有友好提示，而不是一片空白？
   - **异常容错**：API 请求是否有 try/catch？失败时是否有 UI 反馈（Toast/弹窗）？网络超时是否有处理？
   - **状态流转逻辑**：是否存在不合逻辑的跳转或状态回退？状态机是否只能单向流转？
   - **数据闭环**：表单提交后是否重置/刷新？创建数据后列表是否同步？删除后分页是否正常？
   - **并发与防抖**：按钮是否有 loading 防重复点击？轮询是否在页面销毁时清除？
   - **一致性**：前后端对同一概念的命名/枚举值是否完全一致？HTTP状态码处理是否覆盖后端所有可能返回值？
// turbo
6. 将发现的每一个问题记录到 `todo_list`，格式为：`[P级别][Code-Bug/Logic-Bug][文件名] 问题描述`
   - `[Code-Bug]`：语法错误、引用错误、参数不匹配等代码层面问题
   - `[Logic-Bug]`：业务逻辑缺失、权限漏洞、边界未处理等逻辑层面问题（如缺少防抖，直接视为 Bug）
// turbo
7. 该文件审查完毕后，在 `todo_list` 中将对应的审查项标记为 `completed`
// turbo
8. 重复步骤 4-7，直到 **所有文件全部审查完毕**

---

## 第二轮：全量修复（全自动，不停顿）

**原则：发现一个修一个，不询问用户，不等待确认，直接用 edit/multi_edit 工具修改代码**

// turbo
9. 从 `todo_list` 中取出第一个 `pending` 状态的 bug 项，标记为 `in_progress`
// turbo
10. 使用 `read_file` 阅读相关文件的完整上下文
// turbo
11. **立即使用** `edit` 或 `multi_edit` 工具进行 **最小化精准修复**，遵循以下原则：
    - 优先修复上游根因，而非下游打补丁
    - 不引入新功能，不改变无关代码
    - 保持现有代码风格一致
    - **不要停下来问用户"是否需要修复"，直接修**
    - **业务逻辑修复策略**（对 `[Logic-Bug]` 类问题，若无明确文档则按行业最佳实践自动决策）：
      - *输入无校验* → 添加前端+后端双重验证（min/max/类型检查）
      - *接口报错无提示* → 添加用户友好的错误反馈
      - *敏感操作无权限守卫* → 添加角色/登录状态判断
      - *按钮无防重复* → 添加 loading 状态锁
      - *页面销毁未清理* → 添加定时器/监听器清理逻辑
// turbo
12. 修复完成后，将该 bug 项标记为 `completed`
// turbo
13. **立即** 重复步骤 9-12，直到所有 bug 项全部修复完毕。**不要在中间输出总结或等待用户回复。**

---

## 第三轮：验证（浏览器或代码级推演）

> 注意：如果项目不支持浏览器验证（如微信小程序、CLI工具），则跳过浏览器操作，改为 **代码级逻辑推演验证**（走读关键路径的调用链，确认数据流完整闭环）。

// turbo
14. 使用浏览器工具导航到项目页面（若适用）；或使用代码搜索工具走读关键路径
// turbo
15. **正向验证** — 逐个验证已修复的功能：
    - 点击相关按钮/操作，检查功能是否正常
    - 检查浏览器控制台是否有报错
    - 检查数据是否正确更新（UI数值、状态变化）
// turbo
16. **对抗性验证**（找茬测试）：
    - 边界输入：代码是否能防住 `-1`、`0`、空字符串、超长字符串？
    - 防重复：快速连续触发同一操作，是否有 loading 锁？
    - 权限绕过：未登录/非授权用户能否访问受保护的路由或接口？
    - 分页边界：删除列表最后一条数据后，翻页逻辑是否正常？
    - 断线/超时：网络异常时 UI 是否有反馈而非静默失败？
// turbo
17. 如果发现新问题，添加到 `todo_list` 并 **立即回到步骤 9 继续修复，不询问用户**

---

## 循环判定（自动执行，不停顿）

// turbo
18. 检查 `todo_list` 中是否还有 `pending` 或 `in_progress` 状态的项目
// turbo
19. **如果有**：立即回到步骤 9 继续修复，不输出中间报告
// turbo
20. **如果没有**：执行一次快速回归验证（步骤 14-17）
// turbo
21. **如果回归验证也通过**：输出最终审查报告，工作流结束

## 注意事项

- **全程自动执行**：审查→修复→验证的完整循环中不停下来问用户，直接做
- **绝不跳过任何文件**：即使文件看起来"不重要"，也必须完整阅读
- **绝不假设代码正确**：必须实际阅读代码验证，不能凭文件名或注释判断
- **每个问题都要修复**：不能标记为"已知问题"然后跳过
- **修复后必须验证**：不能只改代码不测试
- **循环直到零缺陷**：只要还有问题就继续循环，不设置最大轮数限制
- **不要在修复过程中输出"是否需要修复？"之类的确认请求**
- **业务逻辑缺失等同于 Bug**：缺少输入校验、缺少权限检查、缺少空状态提示，均视为必须修复的缺陷
- **无文档时按最佳实践决策**：对于业务逻辑类问题，若无明确需求文档，按行业通用标准自动修复（如：数量不可为负、删除需确认、敏感操作需鉴权）
- **适配不同项目类型**：若项目为小程序/CLI/桌面应用等非浏览器项目，验证阶段改为代码级逻辑推演
