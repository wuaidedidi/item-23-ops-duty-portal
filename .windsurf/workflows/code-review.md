# 代码审查

对代码进行系统性审查，发现问题按严重性分级。

## 步骤

1. **确定审查范围**：
   - 如果用户指定了文件，审查该文件
   - 如果未指定，用 `find_by_name` 扫描所有源码文件

2. **逐文件审查**：对每个文件用 `read_file` 完整阅读，检查：

   **P0 致命问题**（必须立即修复）：
   - 运行时崩溃（未定义变量、空指针、死循环）
   - 数据丢失风险（存档损坏、状态不同步）
   - 安全漏洞（XSS、注入）

   **P1 严重问题**（应该修复）：
   - 功能未实装（空函数、TODO占位、事件无监听）
   - 逻辑错误（条件反转、边界溢出、类型错误）
   - 接口不匹配（事件名不一致、参数缺失）

   **P2 一般问题**（建议修复）：
   - 代码重复（可抽取公共函数）
   - 命名不清晰
   - 缺少错误处理
   - 性能隐患

   **P3 建议**（可选）：
   - 代码风格不一致
   - 可读性改进
   - 架构优化建议

3. **输出审查报告**：

   ```
   # 代码审查报告

   ## P0 致命问题（X 个）
   | 文件 | 行号 | 问题 |
   |------|------|------|

   ## P1 严重问题（X 个）
   | 文件 | 行号 | 问题 |
   |------|------|------|

   ## P2 一般问题（X 个）
   | 文件 | 行号 | 问题 |
   |------|------|------|

   ## 总结
   - 审查文件数：X
   - 问题总数：X（P0: X, P1: X, P2: X）
   ```

4. **P0/P1 问题阻塞**：如果存在 P0 或 P1 问题，提示用户是否立即修复。确认后进入 `/execute` 工作流修复。
