# 测试驱动开发（TDD）

严格遵循 RED → GREEN → REFACTOR 循环。

## 步骤

1. **RED - 写失败测试**：
   - 根据需求编写一个测试用例
   - 测试应该描述期望的行为，而非实现细节
   - 运行测试，确认它**失败**（红色）
   - 如果测试直接通过，说明测试写得不对或功能已存在

2. **GREEN - 写最小实现**：
   - 只写刚好能让测试通过的代码
   - 不要写"顺便"的代码
   - 不要优化，不要重构
   - 运行测试，确认它**通过**（绿色）

3. **REFACTOR - 重构**：
   - 测试通过后，审视代码质量
   - 消除重复，改善命名，提取函数
   - 每次重构后运行测试确保仍然通过
   - 不改变行为，只改善结构

4. **循环**：回到步骤 1，写下一个测试

## 适用场景

- 用户明确要求 TDD
- 构建核心业务逻辑（计算、状态机、数据转换）
- 修复 bug 时先用测试复现

## 不适用场景

- 纯 UI 布局调整
- 配置文件修改
- 一次性脚本

## 原则

- **测试先于代码** — 绝不在没有测试的情况下写实现
- **一次一个测试** — 不要批量写测试
- **测试要具体** — 用具体的输入输出，不用模糊断言
- **不删测试** — 除非用户明确要求，否则永远不删除或弱化现有测试
